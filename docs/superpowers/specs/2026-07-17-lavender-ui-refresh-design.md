# 温柔薰衣草全项目 UI 优化设计

## 目标

在不修改任务、家庭、iPad 使用管理和 Supabase 数据逻辑的前提下，统一项目的视觉语言、页面层级、弹窗结构和移动端体验。设计应让家庭成员（包括儿童）快速理解当前状态，同时让管理操作保持克制、可靠和可访问。

本轮采用已确认的 A 方案「温柔薰衣草」。现有紫色品牌识别继续保留，但减少厚重渐变、随机阴影、过度装饰和不一致的圆角。

## 范围

### 包含

- 首页顶部成员、激励文案和管理入口。
- 每日任务日历、日期状态和当天任务列表。
- 任务管理、任务类型管理、添加与编辑任务。
- iPad 使用管理标题、额度概览、使用记录、月历和类型管理。
- 成员选择、加入家庭、成员管理和头像修改。
- 密码验证、修改管理密码。
- 完成记录、任务详情、撤销完成和删除确认。
- 桌面、平板和手机响应式布局。
- 焦点、悬停、按下、禁用、加载、成功和错误状态。

### 不包含

- 不修改任务与 iPad 使用业务规则。
- 不修改 Supabase 表、RPC、同步逻辑或图片上传规则。
- 不增加新页面、新导航层级或暗色模式。
- 不引入 UI 框架、图标依赖或构建工具。

## 视觉系统

设计系统以 `design-system/default/MASTER.md` 为基础，结合项目现有紫色资产做以下落地约束。

### 色彩

- 页面背景：浅薰衣草与暖白之间的低对比渐变，避免纯白大面积刺眼。
- 内容表面：白色卡片；次级表面使用轻微紫灰色。
- 主色：沉稳薰衣草紫，用于主要操作、当前日期和键盘焦点。
- 成功色：深绿色，用于完成勾选、正常额度和成功提示。
- 警告色：琥珀色，用于接近额度或长时间使用。
- 危险色：可读性足够的红色，用于超时、删除和错误。
- 文字：深紫黑主文字与中性紫灰次文字，正文对比度不低于 WCAG AA。
- 状态不能只依赖颜色，必须同时提供图标、文字或形状。

### 字体与节奏

- 使用系统圆体栈，避免引入阻塞加载的远程字体。
- 正文移动端不低于 16px；辅助说明不低于 12px。
- 标题采用 18 / 22 / 28px 层级，字重 650–750。
- 使用 4px 基础、8px 主节奏的间距体系。
- 页面内容最大宽度统一为 1240px；卡片圆角统一为 20–22px；控件圆角为 12–14px。

### 阴影与动效

- 卡片只使用一个低对比软阴影层级。
- 弹窗使用统一高层级阴影和 52% 左右的深色蒙层，并轻微模糊背景。
- 微交互时长 150–240ms，只动画透明度、阴影和 transform。
- 支持 `prefers-reduced-motion`，关闭非必要动画。
- 按钮按下反馈不能引发布局抖动。

## 页面结构

### 首页

- 顶部一行由成员资料、居中激励文案和右侧操作组成。
- 操作入口继续使用紧凑图标，但每个点击区域至少 44×44px，并保留 `aria-label` 与提示文字。
- 桌面端日历与任务同排，日历略宽；两张卡片顶部和内容边界对齐。
- 手机端依次展示成员与入口、激励文案、完整日历、当天任务，不出现横向滚动。
- 日期状态统一为：有任务短横、全部完成绿色勾、未完成错误标记、未来任务不判定完成状态。

### 任务列表与任务管理

- 任务行采用轻量次级表面，将完成控件、标题信息和类型标签分成稳定三列。
- 仅圆形完成控件触发完成或撤销；标题区域仍用于查看详情。
- 管理弹窗顶部只保留“任务列表”、次级类型入口、主要添加按钮和弱化关闭按钮。
- 编辑、删除入口保持可发现，但不与任务标题竞争。

### iPad 使用管理

- 独立页顶部只显示返回、居中标题和核心添加入口。
- 父额度创建状态使用清晰引导卡；创建后显示额度、已用、剩余和记录数四项概览。
- 桌面端使用记录与月历同排，记录区更宽；手机端月历完整展示并位于记录列表上方。
- 记录行稳定展示内容、类型、时间和本次时长。进行中计时使用等宽数字，避免刷新抖动。
- 45 分钟黄色、60 分钟红色；不计额度类型使用文字标签和弱化表面。
- 当天超时同时在概览、记录和日历上表达，不只依赖红色。

## 弹窗与表单

### 统一结构

所有弹窗采用四段式结构：标题与关闭、必要说明、内容区、底部操作区。

- 桌面端居中，常规最大宽度 560–620px；任务管理可使用更宽弹窗。
- 手机端采用有页面边距的底部弹层，不全屏，并保留关闭按钮和安全区。
- 关闭按钮使用 44×44px 点击区域，图标严格居中，视觉上弱于主要操作。
- 主操作每个弹窗只保留一个；取消使用文本或次级按钮。
- 危险操作与普通操作分离，并明确说明不可恢复的后果。

### 表单

- 所有输入项提供可见标签；占位符只提供示例，不承担标签职责。
- 必填字段明确标识；复杂选项提供常驻辅助文字。
- 错误显示在对应字段附近并使用 `role="alert"`；提交失败后聚焦首个错误字段。
- 异步提交期间禁用按钮并显示“处理中…”，防止重复提交。
- 密码输入增加显示/隐藏能力，同时保留正确的 `autocomplete` 与数字键盘。
- 图片选择后保留预览与文件处理反馈。

## 响应式与无障碍

- 以 375px 为手机基准，补充 768px、1024px 和 1440px 检查。
- 所有交互目标至少 44×44px，相邻目标间距不少于 8px。
- 不禁用浏览器缩放，不出现固定宽度导致的横向滚动。
- 键盘焦点清晰，Tab 顺序与视觉顺序一致；图标按钮具有可读名称。
- 动态提示使用现有 `aria-live`，弹窗关闭路径清晰。
- 图片预览提供准确替代文本，装饰元素从无障碍树隐藏。

## 实现边界

- 通过整理现有 CSS 文件、增加语义设计令牌和少量 HTML 结构类完成。
- 保持现有 DOM 标识和 JavaScript 查询接口，避免破坏业务逻辑。
- 仅在表单标签、加载状态或无障碍属性确有需要时调整 JavaScript。
- 更新所有静态资源的统一缓存版本号，避免手机继续显示旧样式。

## 验证

- Node 测试覆盖关键 DOM、缓存版本、触控尺寸、焦点和减少动效规则。
- 执行 JavaScript 语法检查和 `git diff --check`。
- 本地浏览器验证首页、任务管理、iPad 页和主要弹窗。
- 视口至少覆盖 375×812、768×1024、1440×900，并检查横竖屏关键布局。
- 验证键盘操作、蒙层、关闭路径、表单错误、图片预览和计时器稳定性。
